# Enterprise Detection Engineering & Threat Hunting Lab

> A production-inspired Detection Engineering and Threat Hunting lab built using Splunk Enterprise, Sysmon, Windows Event Logs, and Windows Security Monitoring to simulate real-world SOC detection, threat hunting, and incident investigation.

---

## Project Overview

The **Enterprise Detection Engineering & Threat Hunting Lab** is a hands-on cybersecurity project designed to demonstrate practical Detection Engineering and Security Operations Center (SOC) skills.

The lab focuses on collecting Windows telemetry using **Sysmon**, forwarding logs to **Splunk Enterprise**, developing detection logic with **SPL (Search Processing Language)**, and building security dashboards that help identify suspicious process execution, command-line activity, registry modifications, PowerShell usage, network connections, and other endpoint behaviors.

Rather than generating alerts only, the project emphasizes how SOC analysts investigate endpoint activity, correlate events, perform threat hunting, and visualize attacker behavior through dashboards.

- Collect Windows endpoint telemetry using Sysmon

# 📊 Dashboard Overview

## Sysmon Threat Detection & Process Monitoring Dashboard

The **Sysmon Threat Detection & Process Monitoring Dashboard** was developed in Splunk Enterprise to provide real-time visibility into Windows endpoint activity using Sysmon telemetry. The dashboard enables SOC analysts to monitor process execution, analyze parent-child process relationships, investigate command-line activity, detect registry modifications, observe network connections, and identify suspicious behavior through interactive visualizations.

This dashboard simulates real-world SOC monitoring by transforming raw Windows event logs into actionable security insights for threat detection and investigation.

---

## Dashboard Panels

| Panel | Description |
|--------|-------------|
| **Total Processes Created** | Displays the total number of Sysmon Process Creation (Event ID 1) events collected during the selected time range. |
| **Top 10 Executed Processes** | Shows the most frequently executed processes to establish normal system activity and identify anomalies. |
| **Top Parent Processes** | Displays parent processes that launch child processes, helping analysts investigate process relationships. |
| **PowerShell Execution Monitoring** | Monitors PowerShell executions, including user, parent process, command line, and integrity level. |
| **Command Prompt Execution Monitoring** | Tracks Command Prompt (cmd.exe) executions for investigation of interactive and scripted command activity. |
| **Network Connection Monitoring** | Displays outbound network connections generated by Windows processes using Sysmon Event ID 3. |
| **Registry Modification Monitoring** | Detects registry modifications captured through Sysmon Event ID 13 to identify persistence-related activities. |
| **High Integrity Process Monitoring** | Identifies processes running with High or System integrity levels, indicating elevated privileges. |
| **Suspicious Process Detection** | Highlights execution of commonly abused Windows binaries such as PowerShell, CMD, Rundll32, Regsvr32, MSHTA, Certutil, and WMIC. |
| **Process Creation Timeline** | Visualizes process creation events over time to help detect spikes, unusual activity, and attack patterns. |

# 🏗️ Lab Architecture

The lab simulates a real-world Security Operations Center (SOC) environment where endpoint telemetry is collected, forwarded, indexed, and analyzed for threat detection.

The environment consists of a Windows 10 virtual machine running Sysmon, which generates detailed endpoint telemetry. Splunk Universal Forwarder collects these events and forwards them to Splunk Enterprise running on the host machine. Detection logic is developed using SPL (Search Processing Language), and the results are visualized through a custom SOC dashboard.

---

## Data Flow

1. User activity is performed on the Windows 10 virtual machine.
2. Sysmon records endpoint events.
3. Splunk Universal Forwarder forwards the events.
4. Splunk Enterprise indexes the data.
5. SPL queries detect suspicious activities.
6. Dashboard panels visualize the results.
7. SOC analysts investigate suspicious events using the collected telemetry.

# 🚨 Incident Response Workflow

When suspicious activity is detected, the following investigation workflow can be followed:

1. Identify the triggered detection.
2. Review the associated process details.
3. Verify the parent-child process relationship.
4. Analyze the command line used.
5. Review user context and privilege level.
6. Investigate related network connections.
7. Examine registry modifications (if applicable).
8. Determine whether the activity is legitimate or malicious.
9. Document the findings.
10. Escalate the incident if malicious activity is confirmed.
